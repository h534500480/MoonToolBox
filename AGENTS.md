# 通用仓库协作规则

## 文档目的

本文件用于约束 Codex 在当前仓库中的默认工作方式，确保后续会话进入仓库后，能够在合理范围内获取上下文、按统一工程规范修改代码，并在输出中清楚说明结论、影响与未验证项。

## 适用范围

1. 本规则面向通用软件仓库，不绑定特定语言、框架、平台或业务域。
2. 若仓库内存在更细分的目录级规范、模块级说明或技术栈专项文档，优先遵守更具体的规则。
3. 若用户明确提出与本文件冲突的要求，以用户要求为准，但需要明确指出偏离点及可能影响。

## 扫描与上下文规则

1. 只要开始做目录扫描、文件搜索、项目遍历或建立仓库结构理解，必须先检查 `.codexignore`。
2. 若 `.codexignore` 存在，则扫描范围必须先按其过滤，不要先全仓扫描再筛选。
3. 若任务确实需要查看被 `.codexignore` 过滤掉的目录，必须先说明原因，再进入这些路径。
4. 默认优先关注以下内容：
   - 根目录说明文件，如 `README`、`AGENTS.md`、设计说明、变更说明
   - 根目录构建、启动、测试、部署脚本
   - 主业务目录、核心模块目录、公共库目录
   - 配置文件、依赖清单、CI 配置、代码风格配置
5. 以下内容默认不作为优先关注对象，除非任务明确涉及：
   - 第三方依赖目录
   - 构建产物目录
   - 缓存目录
   - 大体积数据目录
   - 自动生成代码目录
   - 二进制、压缩包、日志、导出文件目录

## 环境与执行边界

1. 不要假设当前工作区一定具备完整运行、构建、测试环境，先依据仓库说明、脚本和现有配置判断。
2. 对构建、测试、代码生成、数据库迁移、基础设施变更等动作，优先选择与当前环境兼容、风险可控的验证方式。
3. 若命令依赖特定操作系统、容器、硬件、云资源、私有服务或外部凭证，执行前必须明确说明限制。
4. 未实际运行、未构建通过、未测试通过或未在目标环境验证的改动，必须明确写出“尚未验证”。
5. 除非用户明确要求，否则不要擅自执行以下高影响动作：
   - 推送远端
   - 改写历史
   - 发布、部署、上线
   - 删除数据库、清空缓存、覆盖生成大批文件

## 代码输出规则

1. AI 新增或重写的代码文件，文件头部必须有简洁的功能描述注释。
2. 所有新增代码注释必须使用中文。
3. 注释内容必须专业、克制、可维护，不得加入 emoji、颜文字、口号式表述或无信息量描述。
4. 重要函数、类、接口、回调、状态机、任务编排入口必须写清楚：
   - 功能说明
   - 输入输出或关键依赖
   - 用法说明或注意事项
5. 注释要解释“为什么这样做”或“这里要注意什么”，不要复述显而易见的代码行为。
6. 生成代码时要优先保证结构清晰，不允许通过无脑叠加 `if/else`、超长函数、超深层嵌套来拼出功能。
7. 遇到复杂分支时，优先考虑以下方式降低复杂度：
   - 提前返回
   - 拆分职责单一的辅助函数
   - 使用清晰的策略映射、状态分发或配置驱动
   - 通过类型、接口或数据结构表达约束，而不是堆砌条件判断
8. 新增代码应尽量满足以下可读性要求：
   - 单个函数职责明确
   - 命名直观，避免含糊缩写
   - 控制流可顺着读懂
   - 异常路径、边界条件、默认值处理明确
9. 修改已有代码时，优先延续原有架构与风格；若原实现明显失衡，可在说明原因后做小范围结构化调整。
10. 未经必要论证，不要引入与当前复杂度不匹配的抽象层、设计模式或过度封装。

## 通用代码规范补充

1. 优先消除重复逻辑，避免复制粘贴式修补。
2. 公共逻辑应沉淀到合适层级，避免业务代码直接散落在入口、控制器、路由、回调或脚本主流程中。
3. 配置、常量、路径、魔法值应集中管理，避免硬编码分散。
4. 错误处理要明确，不能吞异常、静默失败或只打印无上下文日志。
5. 日志、提示、报错文案应可定位问题，至少包含必要上下文。
6. 输入校验、空值处理、边界处理应放在靠近入口的位置，不要把非法状态传到深层逻辑。
7. 接口、函数、模块之间的依赖方向要稳定，避免循环依赖和跨层直接访问。
8. 变更数据结构、协议字段、接口签名、配置项时，必须同步评估兼容性影响。
9. 能用已有测试、脚本、静态检查覆盖的改动，应优先补齐或更新相应验证。
10. 生成示例项目或初始化工程时，目录结构要简洁，职责划分要清楚，不要生成表面完整但难以维护的模板代码。

## 目录结构与模块分层规则

1. 创建新项目、初始化模块或重构目录时，必须先确认整体框架，再按职责划分目录层级，不允许把不同性质的功能长期堆在同一目录下。
2. 目录结构应做到语义清晰、层级稳定、便于定位，看到目录名后应能大致判断其中代码的职责范围。
3. 目录划分方式应根据项目规模、语言生态、框架约定和业务复杂度决定，不强制套用固定分层模板。
4. 同一层级目录应尽量保持划分维度一致，不要同时混用“按技术类型拆分”和“按业务领域拆分”而没有明确边界。
5. 项目较小时，可以保持简洁结构；项目变复杂后，应及时把不同职责拆开，避免所有功能长期堆在同一目录。
6. 模块划分的核心要求是职责清晰、边界明确，至少要避免把入口逻辑、业务逻辑、数据访问、外部依赖适配和通用工具长期混写。
7. `common`、`utils`、`helpers`、`shared` 一类目录只能放通用且稳定的能力，不得把临时业务逻辑、特殊分支逻辑或模块私有实现长期塞入其中。
8. 跨模块依赖应尽量单向，公共模块不应反向依赖具体业务模块。
9. 若新增目录或拆分模块，必须说明：
   - 新目录负责什么
   - 为什么原位置不再合适
   - 与相邻目录的边界是什么
10. 生成项目脚手架时，默认应包含最小但清晰的目录骨架，例如：
   - 源码目录
   - 测试目录
   - 配置目录
   - 脚本目录
   - 文档目录
   - 构建或部署相关目录
11. 不要为了“看起来规范”而制造过多空目录或过深层级；目录分层的目标是降低理解成本和修改风险，而不是增加形式复杂度。
12. 若当前仓库目录已经混乱，新增改动时应优先保证新代码放在正确位置；若顺手做结构整理，必须控制范围，避免把任务扩大成大规模搬迁。

## 注释与文档规则

1. 只在确有必要的地方补注释，不为每一行或每个简单变量添加注释。
2. 文档、注释、说明中的术语应前后一致，避免同一概念多种叫法。
3. 若修改影响使用方式、启动方式、配置方式、接口行为或调试手段，应同步更新相关文档。
4. 若只是文档或注释修改，也要说明为什么要改、解决了什么理解成本。

## 项目生成与 `.gitignore` 规则

1. 若任务涉及创建新项目、生成脚手架或补充仓库基础文件，默认应创建或更新 `.gitignore`。
2. `.gitignore` 至少应覆盖以下内容，并结合实际技术栈补充：
   - 操作系统垃圾文件，如 `.DS_Store`、`Thumbs.db`
   - 编辑器与 IDE 目录，如 `.vscode/`、`.idea/`
   - 临时文件、缓存目录、日志目录
   - 构建产物目录、测试产物目录、覆盖率目录
   - 依赖下载目录、虚拟环境目录、包管理缓存目录
   - 工程生成文件、锁文件以外的本地状态文件
   - 与具体工程相关的大文件、导出文件、中间文件、私有配置文件
3. 若仓库属于特定技术栈，应补充对应忽略项，例如：
   - Python: `__pycache__/`、`.pytest_cache/`、`.mypy_cache/`、`.venv/`
   - Node.js: `node_modules/`、`.pnpm-store/`、`.next/`、`dist/`
   - Java: `target/`、`.gradle/`、`build/`
   - C/C++: `build/`、`out/`、`bin/`、`obj/`
   - Rust: `target/`
   - Go: `bin/`、覆盖率文件
4. 若项目有明确应纳入版本控制的示例配置文件，应使用如 `.env.example` 的形式保留模板，而不是忽略全部配置文件。
5. 新建 `.gitignore` 或新增忽略项时，要说明新增项对应的是哪类缓存、产物或工程文件，以及这样处理的原因。

## 修改说明规则

1. 每次修改完成后，必须明确说明修改了哪些文件、哪些部分。
2. 若修改了参数、配置项、环境变量、接口、脚本入口或目录结构，必须说明：
   - 原本作用
   - 为什么要改
   - 预期影响是什么
3. 若修改前后会导致行为变化，必须说明调用方式、执行路径、输出结果或兼容性上的变化。
4. 若存在未确认事实，必须明确标注为“推断”还是“已确认”。

## 提交与分支规则

1. 不要擅自推送远端。
2. 不要擅自改写历史，除非用户明确要求。
3. 若用户要求提交代码，提交说明优先使用简洁中文。
4. 提交前应尽量说明本次提交包含哪些文件及主要改动点。

## 输出风格规则

1. 回答要直接、清晰、务实。
2. 优先说明结论，再说明原因和影响。
3. 不要堆砌无关背景，不要把显而易见的过程写成冗长报告。
4. 说明实现时，优先描述结构、职责边界、关键数据流或关键调用链，而不是只罗列语法细节。

## 建议后续可继续补充的规则

1. 为不同类型文件统一头部注释模板，例如：
   - `cpp`
   - `py`
   - `ts`
   - `java`
   - `yaml`
   - `Dockerfile`
2. 对重要参数修改增加“旧值 / 新值 / 默认值 / 兼容性”说明要求。
3. 对不同技术栈补充固定检查清单，例如构建、单测、静态检查、格式化、发布前检查。
4. 对接口变更增加固定的影响评估模板，例如调用方、兼容策略、回滚方式。
